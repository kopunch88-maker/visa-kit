# services/ — бизнес-логика

В этой папке — слой между API и моделями. Сюда выносится всё, что:
- Сложнее одного запроса к БД
- Использует внешние сервисы (LLM, OCR, ЦБ РФ)
- Может вызываться из нескольких эндпоинтов (рендеринг, валидация)

## Файлы

- `recommendation.py` — LLM-рекомендация компании/должности по CV
- `passport_ocr.py` — распознавание паспорта через Anthropic Vision
- `cbr_rates.py` — курс ЦБ РФ для конвертации RUB→EUR
- `rendering.py` — генерация пакета документов из шаблонов
- `status_machine.py` — переходы между статусами Application
- `monthly_documents.py` — генератор актов/счетов (3 пары на месяц)
- `bank_statement.py` — генератор выписки Альфа-банка с реалистичными транзакциями
- `countries.py` — справочник стран (ISO, отображаемые имена)

## Принципы

### 1. Async везде, где есть IO

LLM, OCR, ЦБ — все делают HTTP-запросы. Используем `httpx.AsyncClient`,
функции async. БД-операции в Session — синхронные (sqlmodel это
синхронный, для простоты). Внутри async-функций используем
`run_in_threadpool` для DB.

### 2. Внешние сервисы — через адаптеры

Никогда не дёргаем `httpx.get(...)` прямо в эндпоинте. Каждый внешний
сервис обёрнут в свой класс/модуль:

- `services/anthropic_client.py` — единая точка для всех вызовов LLM
- `services/cbr_client.py` — единая точка для курсов
- `services/s3_client.py` — единая точка для S3

Эти модули знают про retry, кеширование, ошибки. Эндпоинты их просто
вызывают.

### 3. Кеш для дорогих операций

- Курсы ЦБ — кешируем по дате (один и тот же день → не запрашиваем дважды)
- LLM-рекомендации — НЕ кешируем (это история, должна сохраняться отдельно
  для каждого запроса)
- OCR паспорта — НЕ кешируем (запускается один раз на клиента, потом
  результат уже в БД)

### 4. Тесты обязательны для computed-логики

`bank_statement.py` (генерит реалистичные транзакции с НПД, KWIKPAY и т.д.)
и `monthly_documents.py` (даты актов/счетов) — это самая хрупкая логика.
Один тест в `tests/test_bank_statement.py` спасает от багов в продакшене.
