# models/ — сущности БД и API

В этой папке — все SQLModel модели. Они работают одновременно как:
- ORM-сущности (через SQLAlchemy под капотом)
- Pydantic-схемы (для FastAPI запросов/ответов)
- Источник правды для типизации фронтенда (через OpenAPI)

## Список сущностей

### Справочники (создаются один раз, переиспользуются)
- `Company` — наши юридические лица. 8 шт на старте.
- `Position` — типовые должности с описанием обязанностей и ставкой. Связана с Company.
- `Representative` — наши представители в Испании (NIE, контакты, ЭЦП).
- `SpainAddress` — типовые адреса в Испании, по которым подаёмся.

### Заявка
- `Applicant` — данные кандидата: имя, паспорт, опыт, образование. Заполняет клиент.
- `Application` — связка всего вместе: один Applicant + назначенные Company, Position, Address, Representative + контекст подачи.
- `FamilyMember` — член семьи (для семейных подач, опционально).
- `PreviousResidence` — страна, где жил клиент (для допсправок о несудимости).
- `UploadedFile` — загруженный клиентом или менеджером файл (паспорт, справка, EGRYL).
- `GeneratedDocument` — сгенерированный системой документ (договор, акт и т.п.).

### Системные
- `User` — менеджер команды (для админки).
- `Application.timeline_events` — лог действий по заявке (кто что сделал).

## Главные правила

### 1. Связи через id, не через дублирование

```python
# ❌ Плохо — данные дублируются
class Application(SQLModel):
    company_name: str
    company_inn: str
    company_address: str

# ✅ Хорошо — ссылка на справочник
class Application(SQLModel):
    company_id: int = Field(foreign_key="company.id")
```

### 2. Обновление справочника не трогает старые заявки

Если у Company сменился директор, в **новых** заявках будет новый директор,
а в **уже подписанных** документах — старый. Реализация: при рендере
GeneratedDocument сохраняется снапшот данных компании на момент рендера в
`GeneratedDocument.snapshot_data` (JSONB).

### 3. Миграции — через Alembic

Любое изменение модели → `alembic revision -m "what changed"` → правим
миграцию руками (важно для прода, чтобы данные не потерять) → `alembic upgrade head`.

Никогда не правим существующие миграции — только добавляем новые.

### 4. Поля для опционального гражданства

Не у всех есть ИНН (только россияне). Не у всех адрес в РФ. Это значит:

```python
class Applicant(SQLModel):
    nationality_code: str      # ОБЯЗАТЕЛЬНО, ISO 3-letter (RUS, AZE, KAZ...)
    inn: Optional[str] = None  # только для россиян
    home_address: str          # любой формат, любая страна
    passport_number: str       # любой формат, без regex
```

### 5. Бизнес-валидация — методом, не ORM-валидатором

В моделях — только синтаксическая валидация (типы, длина строк). Все
бизнес-правила (договор ≥90 дней, минимум зарплаты) — в методе
`Application.validate_business_rules()` который вызывается перед рендером.

## Когда добавлять новое поле

1. Открыть `models/<entity>.py`
2. Добавить поле с дефолтом или Optional (чтобы не сломать существующие записи)
3. `cd backend && alembic revision --autogenerate -m "add field X to entity Y"`
4. Проверить сгенерированную миграцию глазами (alembic иногда промахивается)
5. `alembic upgrade head` локально + проверить, что приложение стартует
6. Если поле нужно в шаблонах документов — добавить переменную в шаблон

## Когда добавлять новую сущность

1. Создать файл `models/<new_entity>.py` по образцу `models/company.py`
2. Импортировать в `models/__init__.py`
3. Создать миграцию
4. Создать API-роутер `api/<new_entity>.py` по образцу `api/companies.py`
5. Зарегистрировать роутер в `app/main.py`
6. Регенерить фронтовые типы: `cd frontend && npm run gen:api`
