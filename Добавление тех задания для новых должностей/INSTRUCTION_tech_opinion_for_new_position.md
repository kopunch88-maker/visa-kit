- **Уровень Lead (L4)** — нужна специфика: руководит распределённой командой? Сколько подчинённых? Это влияет на §3 steps и §4 grounds.

## 3. Стилистика контента (КРИТИЧНО)

### Описание (description_ru/es)

- **Длина:** 400-600 символов
- **Формат:** один длинный абзац, элементы через `;`
- **Тон:** деловой, без воды
- **Перечисляются:** реальные функции из `duties`, но переформулированные в формат «делает то-то, делает то-то»
- **Не дублировать** `duties` дословно
- **Ключевое:** упомянуть инструменты/системы, через которые осуществляется работа (СЭД, CRM, BIM, IDE и т.д.) — это готовит почву для §2

**Пример (PR Manager):**
> разработка коммуникационной стратегии компании; подготовка и согласование пресс-релизов, информационных сообщений, корпоративных новостей; ведение коммуникации со СМИ и блогерами по электронной почте; организация интервью и комментариев экспертов компании в формате видеосвязи; мониторинг упоминаний компании в СМИ и социальных сетях…

### Инструменты (tools_ru/es)

- **Количество:** 5-8 шт.
- **Структура:** `[{"name": "...", "purpose": "..."}]`
- **`name`:** название инструмента, можно с альтернативами через `/` («Notion / Confluence»)
- **`purpose`:** что именно делается в этом инструменте (короткая фраза)
- **Обязательно включать в список:**
  - **1-2 универсальных:** Zoom/Google Meet/MS Teams (видеосозвоны), Slack/Telegram (коммуникация)
  - **2-4 профессиональных:** SQL, Power BI, Adobe Photoshop, Trados — что соответствует профессии
  - **1-2 систем хранения:** Confluence, Google Drive, Git, BIM 360

**Пример:**
```json
[
  {"name": "SQL (PostgreSQL, ClickHouse)", "purpose": "запросы к хранилищам данных"},
  {"name": "Power BI / Tableau", "purpose": "построение дашбордов"},
  {"name": "Zoom / Microsoft Teams", "purpose": "коммуникация с заказчиками"}
]
```

### Шаги (steps_ru/es)

- **Количество:** 6-8 шт.
- **Структура:** `[{"title": "...", "body": "..."}]`
- **`title`:** короткое название этапа (3-5 слов)
- **`body`:** 1-2 предложения с описанием, **обязательно упоминающих цифровые средства** («через корпоративный портал», «в Jira», «через видеосозвон», «в облачной среде»)
- **Логика:** end-to-end рабочий процесс от получения задачи до сдачи результата
- **Каждый шаг** должен явно или неявно показывать дистанционность

**Канонические шаги для большинства интеллектуальных профессий:**
1. Получение задачи (через Jira/Notion/email)
2. Анализ исходных данных (электронные документы)
3. Основная работа в специализированном ПО (главный инструмент)
4. Внутреннее согласование (через систему/мессенджер)
5. Передача результата (через облако/ЭДО)
6. Согласование с заказчиком (видеосозвон, демонстрация экрана)
7. Корректировки (по итогам обратной связи)
8. Финальная сдача (защищённое облачное хранилище)

### Основания (grounds_ru/es)

- **Количество:** 2-3 шт.
- **Структура:** просто список строк
- **Каждое основание** — 1-2 предложения, аргументирующие почему профессия совместима с remote-форматом

**Шаблонные паттерны:**
1. «Все инструменты профессии работают на стандартном компьютере…» (про софт)
2. «Документация существует исключительно в электронном виде…» (про артефакты)
3. «Современная практика [отрасли] построена на удалённой работе…» (про индустрию)
4. «Все взаимодействия (с заказчиком, командой, регуляторами) происходят через…» (про коммуникацию)

### Пункт договора (contract_clause_ru/es)

- **1 предложение** в формате: «Услуги по [деятельности] оказываются дистанционно с использованием [инструментов/систем]».
- Пример: «Услуги по сбору, обработке и визуализации данных оказываются исключительно дистанционно с использованием средств электронной связи и облачных хранилищ данных.»

## 4. Алгоритм Claude

Когда Костя присылает запрос «заполни tech_opinion для должности X», я:

### Шаг 1. Понять контекст

- Прочитать `title_ru`, `title_es`, `level`, `duties`, `profile_description`
- Определить **категорию профессии**:
  - **IT-разработка** → использовать `_dev_block`-like контент
  - **Бизнес-аналитика / консалтинг** → `_ba_block`-like
  - **Продажи / клиентский сервис** → `_sales_block`-like
  - **Инженерия (строительство, электрика и т.д.)** → `_engineering_block`-like
  - **Творческие профессии** (дизайн, журналистика, копирайтинг) → кастомный блок
  - **Регулируемые** (юрист, психолог, врач) → кастомный с упоминанием специфики
  - **Преподавание / наука** → онлайн-формат подчёркивать
  - **Управление / Lead** → акцент на дистанционном руководстве распределённой командой

### Шаг 2. Проверить совместимость с remote

Если профессия очно-присутственная по природе (медсестра, повар, водитель, физтерапевт):

- **НЕ ЗАПОЛНЯТЬ** автоматически
- Объяснить Косте: «Эта должность требует физического присутствия по природе работы. Заполнить tech_opinion для неё нечестно — испанский консулат отклонит. Варианты: (а) не делать tech_opinion для этой должности; (б) если есть реальная история удалённой работы в этой профессии — расскажи детали, тогда сделаю специфичный обоснованный текст».

### Шаг 3. Сгенерировать контент

- Длины: description 400-600 симв; 5-8 tools; 6-8 steps; 2-3 grounds; contract_clause 1 предложение
- Стиль: деловой, без эмоций, активный залог
- Инструменты: специфика профессии + универсальные (Zoom/Slack)
- Steps: end-to-end от получения задачи до сдачи
- Grounds: 1 про инструменты, 1 про артефакты, 1 про индустрию
- **Испанские версии** — НЕ дословный перевод; адаптировать под испанские реалии (например, ФНС → AEAT, Госуслуги → Sede Electrónica, СЭД → gestión documental electrónica)

### Шаг 4. Выкатить apply-скрипт

Создать `apply_pack40_0_X_seed_position_N.py` по шаблону из секции 5 ниже.

### Шаг 5. Дать Косте инструкцию


## 5. Шаблон apply-скрипта

> Я (Claude) генерирую этот файл каждый раз для новой Position. Костя сохраняет в `D:\VISA\visa_kit\` и запускает.

```python
"""
Pack 40.0-X — Seed tech_opinion для Position id=N: <title_ru>.
"""

import os
import sys
import psycopg2
from psycopg2.extras import Json

POSITION_ID = N  # <ПОДСТАВИТЬ>

CONTENT = {
    "international_analog_ru": "<...>",
    "international_analog_es": "<...>",
    "description_ru": "<...>",
    "description_es": "<...>",
    "tools_ru": [
        {"name": "<...>", "purpose": "<...>"},
        # 5-8 шт.
    ],
    "tools_es": [
        {"name": "<...>", "purpose": "<...>"},
    ],
    "steps_ru": [
        {"title": "<...>", "body": "<...>"},
        # 6-8 шт.
    ],
    "steps_es": [
        {"title": "<...>", "body": "<...>"},
    ],
    "grounds_ru": [
        "<основание 1>",
        "<основание 2>",
        # 2-3 шт.
    ],
    "grounds_es": [
        "<motivo 1>",
        "<motivo 2>",
    ],
    "contract_clause_ru": "<...>",
    "contract_clause_es": "<...>",
}


def main():
    db_url = os.environ.get("DATABASE_URL")
    if not db_url:
        print("ERROR: DATABASE_URL не задан в env")
        return 1

    conn = psycopg2.connect(db_url)
    cur = conn.cursor()
    try:
        # Проверим что Position существует
        cur.execute("SELECT title_ru FROM position WHERE id = %s", (POSITION_ID,))
        row = cur.fetchone()
        if not row:
            print(f"✗ Position id={POSITION_ID} не найдена")
            return 1
        title_ru = row[0]

        c = CONTENT
        cur.execute(
            """
            UPDATE position SET
                international_analog_ru = %s,
                international_analog_es = %s,
                tech_opinion_description_ru = %s,
                tech_opinion_description_es = %s,
                tech_opinion_tools_ru = %s,
                tech_opinion_tools_es = %s,
                tech_opinion_steps_ru = %s,
                tech_opinion_steps_es = %s,
                tech_opinion_grounds_ru = %s,
                tech_opinion_grounds_es = %s,
                tech_opinion_contract_clause_ru = %s,
                tech_opinion_contract_clause_es = %s
            WHERE id = %s
            """,
            (
                c["international_analog_ru"], c["international_analog_es"],
                c["description_ru"], c["description_es"],
                Json(c["tools_ru"]), Json(c["tools_es"]),
                Json(c["steps_ru"]), Json(c["steps_es"]),
                Json(c["grounds_ru"]), Json(c["grounds_es"]),
                c["contract_clause_ru"], c["contract_clause_es"],
                POSITION_ID,
            ),
        )
        conn.commit()
        print(f"✓ id={POSITION_ID} ({title_ru}): tech_opinion заполнен, rows={cur.rowcount}")
        return 0
    except Exception as e:
        conn.rollback()
        print(f"✗ FAIL: {e}")
        return 1
    finally:
        cur.close()
        conn.close()


if __name__ == "__main__":
    sys.exit(main())
```

## 8. Особые случаи и FAQ

**Q: А если у Position уровень L4 (Lead/Director)?**
A: В `description` подчеркнуть управленческие функции (формирование команды, защита перед руководством, стратегические решения). В `tools` добавить инструменты управленческой отчётности (Power BI, корпоративные дашборды). В `steps` — еженедельные/ежемесячные ритуалы вместо ежедневных. В `grounds` упомянуть что «руководство распределённой командой через интернет — устоявшаяся практика после 2020».

**Q: А если должность имеет физическую составляющую, но 80% времени remote (например, инженер на стройке, врач телемедицины)?**
A: Костя должен явно сказать «оставшиеся 20% не входят в обязанности по данной позиции — это другие сотрудники». Тогда я генерю контент с явной фразой «Полевые/лабораторные/иные физические работы выполняются другими сотрудниками — в обязанности данной должности они НЕ входят» (см. как сделано у Биолога id=68 и Геодезиста id=2).

**Q: Что если новой Position нет в БД?**
A: Костя сначала создаёт её через UI (Settings → Positions → New). Получает `id`. Затем присылает мне `id` и я генерю seed-скрипт. Альтернатива: я генерю SQL `INSERT INTO position (...) VALUES (...) RETURNING id` + сразу UPDATE на tech_opinion — но это сложнее, лучше через UI.

**Q: Куда складывать backup'ы?**
A: Apply-скрипты для tech_opinion НЕ создают backup БД-полей (один UPDATE по id). Если Костя хочет страховку — можно перед запуском сделать дамп: `SELECT title_ru, tech_opinion_description_ru FROM position WHERE id = N;` и сохранить.

**Q: Может ли Костя сам заполнить через UI?**
A: Да, с Pack 41.0 это полностью работает. Я нужен только когда хочется сгенерить контент быстро для новой должности по `duties`.
