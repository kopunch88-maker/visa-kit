# CLAUDE.md — инструкции для LLM-ассистентов

> Этот файл читается Claude Code, Cursor и другими AI-помощниками первым делом
> в каждой сессии. Здесь — главные принципы проекта.

## Контекст проекта

Система генерирует пакет документов на испанскую digital nomad визу. Клиенты
подают через нас — мы оформляем их в наших компаниях (8 юрлиц) на типовых
должностях. Поток 50+ заявок/мес, команда менеджеров + сами клиенты.

Главный enemy — рассинхронизация данных между документами. Если ИНН компании
введён один раз, он должен попасть в договор, акты, счета, выписку, EGRYL и
письмо одинаково. Поэтому архитектура:

```
Анкета клиента → Application (БД) → справочники (Company, Position, Address, Representative)
                ↓
                Шаблоны (DOCX/PDF с переменными {{ }}) ← ОДИН источник истины ← Application
                ↓
                Готовый пакет (~14 файлов)
```

## Главные принципы

### 1. Один источник истины — Application

Любое поле, которое появляется в двух документах, лежит в Application или в
связанном справочнике РОВНО ОДИН РАЗ. Никогда не дублируем.

❌ Плохо: в Application есть `company_inn`, и в Company есть `inn`.
✅ Хорошо: в Application есть `company_id` → ссылка на Company, у которой `inn`.

### 2. Шаблоны — это файлы, а не код

Шаблоны лежат в `/templates/docx/*.docx` и `/templates/pdf/*.pdf`. Это **обычные
документы**, которые менеджер открывает в Word и редактирует. Переменные
обозначены как `{{ contract.salary_rub }}` (Jinja2 синтаксис, обработка docxtpl).

Никогда не зашиваем в код текст, который должен видеть клиент или UGE — он
всегда в шаблоне.

### 3. Бизнес-валидация перед рендером

В `Application.validate_business_rules()` проверяем все правила UGE:
- Договор подписан ≥90 дней назад
- Зарплата в EUR ≥ минимума UGE (текущий минимум см. `docs/business_rules.md`)
- Справки о несудимости не старше 90 дней
- Все файлы загружены

Если проверки не прошли — НЕ рендерим, возвращаем список проблем.

### 4. Переиспользование через справочники

В админке можно один раз создать `Company`, `Position`, `Representative`,
`SpainAddress` — и они переиспользуются во всех будущих заявках.

При создании заявки менеджер не вводит реквизиты компании заново — выбирает
из списка. Это убирает 90% ошибок ввода.

### 5. Поддержка не-РФ клиентов

Не делать жёстких regex для ИНН (12 цифр), адресов («индекс РФ»), паспортов.
ИНН опциональный (только у россиян). Адрес — произвольная строка. Паспорт —
строка любого формата.

Гражданство — обязательное поле, страна юрлица (`Company.country`) — тоже.

### 6. Семейные подачи — ~20% случаев

В Application есть `family_members: List[FamilyMember]`. Если непустой — на
каждого члена семьи генерируется свой суб-пакет (MI-F форма, паспорт,
designación, тасса, банк. сертификат).

## Стиль кода

### Python (backend)

- Black format (4 пробела), line length 100
- Type hints везде, никаких голых dict
- Pydantic/SQLModel для всех данных, никаких dataclasses кроме внутренней логики
- Имена переменных и комментарии — на английском (это стандарт)
- Docstrings ОБЯЗАТЕЛЬНЫ для функций бизнес-логики и API эндпоинтов
- Тесты для computed-слоя ОБЯЗАТЕЛЬНЫ, для остального — желательны

### TypeScript (frontend)

- Strict mode, никаких `any`
- Компоненты — функциональные, с типизированными props
- shadcn/ui компоненты копируем в `components/ui/`, не зависим от внешней библиотеки
- API-клиент в `lib/api.ts` с типизированными ответами (генерим из FastAPI
  OpenAPI через `openapi-typescript`)

## Паттерны для копирования

Когда нужно добавить:

- **Новую сущность справочника** (например, ещё одна Company-like) → копируй
  `app/models/company.py` + `app/api/companies.py` + миграция через alembic
- **Новый шаблон документа** → копируй структуру существующего шаблона + регистрируй
  в `app/templates_engine/registry.py`
- **Новую бизнес-проверку** → добавляй в `Application.validate_business_rules()`
  + тест в `tests/test_validation.py`
- **Новое поле в существующей сущности** → правь модель + создавай миграцию
  alembic + обновляй фронтовый тип через openapi-typescript

## Что НЕ делаем

- Не пишем raw SQL — только через SQLModel (исключения в `db/raw_queries.py`
  если действительно нужно)
- Не сохраняем файлы клиентов на ФС сервера — только в S3
- Не отправляем persondata в LLM без согласия клиента (используем только в
  scope конкретной задачи: OCR паспорта по запросу, рекомендация по CV)
- Не храним пароли в БД (используем magic-link auth для команды)
- Не делаем синхронные вызовы LLM/OCR в HTTP-handler — через background tasks

## Полезные команды

```bash
# Backend
cd backend
pytest                     # тесты
alembic revision -m "..."  # новая миграция
alembic upgrade head       # применить миграции
black . && ruff check .    # форматирование

# Frontend
cd frontend
npm run dev                # dev сервер
npm run build              # production build
npm run gen:api            # перегенерация типов из OpenAPI

# Локальное окружение
cd infra
docker compose up -d       # поднять PG + MinIO
docker compose logs -f     # логи
```
